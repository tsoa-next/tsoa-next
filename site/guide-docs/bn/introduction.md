# ভূমিকা

`tsoa-next` এটাই মূল নিয়ম [`tsoa`](https://github.com/lukeautry/tsoa) সেখানে লুক অট্রি এবং প্রদায়কদের প্রতিষ্ঠিত স্থায়ী ভিত্তি স্থাপন করা হবে।
এটি একটি সমন্বিত কাঠামো OpenAPI কম্পাইল করার সামগ্রী Node.js অ্যাপ্লিকেশন সার্ভার, পরিচিত সার্ভার TypeScript. .
এটা প্রকাশ করতে পারে, কোই, কোরা আর আরো অনেক কাঠামোতে।
`tsoa-next` অ্যাপ্লিকেশনগুলি দ্বারা ডিফল্টরূপে ও বিশেষ ধরনের কর্ম সমর্থিত হয়।

নিচের দিকে, `tsoa` সাধারণত ব্যাখ্যা করা হয় CLI যে সমস্ত স্থাপত্যের ওপর ভিত্তি করে কমান্ড `tsoa-next` চলছে।

## গোল

- TypeScript আপনার API এর একমাত্র সত্যের সোর্স হিসাবে নিয়ন্ত্রক এবং মডেল হিসাবে নিয়ন্ত্রক
- একটি বৈধ OpenAPI (পর্যন্ত) SwaggerDirectors (2.0 বা ৩) এর মাধ্যমে আপনার কন্ট্রোল এবং মডেল থেকে তৈরি:
  - পাথ (যেমন, / etc/ users)
  - ভিত্তির উপর ভিত্তি করে সংজ্ঞা TypeScript ইন্টারফেস ( সমূহ)
  - আবশ্যক অথবা ঐচ্ছিক বৈশিষ্ট্যের উপর ভিত্তি করে পরামিতির মান TypeScript (. মধ্যে উপস্থিত তথ্যগুলি ঐচ্ছিক OpenAPI বর্ণনা
  - বস্তুর বর্ণনার জন্য jDoc সমর্থিত (অধিকাংশ মিটা-ডাটার সম্পূর্ণ হতে পারে) TypeScript ধরন
- বাছাই করার ক্ষেত্রে রুট তৈরী করা হয়
  - Express'%s' Hapiএবং Koa বর্তমানে সমর্থিত GNOME-র সাহায্যে অন্যান্য মাঝের প্রোগ্রাম দ্বারা সমর্থিত হয়
  - সমুদ্রবিহীন প্রসব বৈধ

## পরিবর্তীত

- পুনরায় করা হবে TypeScript যদি সম্ভব হয় তবে API মিটা-ডাটা উত্‍পাদন করো
- রেগুলার এক্সপ্রেশন-র সাথে সম্পূর্ণভাবে টিকা ব্যবহার করা হলে, মিটা-ডাটার ব্যবহার উপযোগী নয়
- টেক্সট মিটা-ডাটার জন্য jisdoc ব্যবহার করুন (যেমন, %s)
- ছোট করুন
- ইন্টারফেসের মাধ্যমে মডেলগুলো সবচেয়ে ভাল (পুর তথ্য কাঠামো), কিন্তু ক্লাসগুলি দ্বারা চিহ্নিত হতে পারে
- সঞ্চালনার বৈধ কর্ম নয় `tsoa-next` যে সমস্ত মান অপরিবর্তিত রাখা হবে সেগুলির ক্ষেত্রে সুনির্দিষ্ট ভাবে প্রয়োগ করা হবে OpenAPI ২/৩ স্কিমা বর্ণনা করেছে। যুক্তির মধ্যে যে পার্থক্য রয়েছে তা প্রজন্মের পর প্রজন্মকে জানিয়ে দেওয়া OpenAPI (ওএস) এবং/অথবা রুট।
  - দয়া করে নোট এটি সক্রিয় করুন OpenAPI ৩ আপনি অবৈধ যুক্তির সুযোগ কমিয়ে দেন OpenAPI ৩ এর আরো বড় কোন যুক্তি আছে।
